## Context

Issue #219 established a three-layer defense model for memory management:

1. **Global `~/.claude/CLAUDE.md`** ā€” authoritative policy (already in place)
2. **Per-repo `CLAUDE.md` snippet** ā€” second line of defense
3. **`MEMORY.md` policy header** ā€” point-of-write trap

This spec covers the fleet rollout that plants layers 2 and 3 across all
consuming repos. It also creates follow-up audit issues for repos with
existing memory files.

## Per-Repo Changes

### 1. MEMORY.md policy header

Plant the canonical HTML comment at the top of `MEMORY.md`. If the file
exists, preserve all index entries below the header. If it does not exist,
create the directory and file.

**Canonical header:**

```text
<!-- POLICY: Do not edit this file or any file in this directory without
explicit human approval. Before writing, propose the content to the human
and suggest where it belongs: repo memory, global CLAUDE.md, or a
plugin/skill issue. See the global CLAUDE.md memory management policy
for the full workflow. User-invoked skills (e.g., /handoff) are exempt. -->
```

**Path resolution:** MEMORY.md lives at
`~/.claude/projects/-Users-pmoore-dev-github-<repo-name>/memory/MEMORY.md`.
All repos are cloned under `/Users/pmoore/dev/github/`. The slug is derived
by replacing `/` with `-` in the absolute path and prepending `-`.

If the slug derivation produces an incorrect path, the result is an orphan
MEMORY.md that does not get used ā€” low blast radius, catchable in audit.

### 2. CLAUDE.md memory management snippet

Add or replace the memory management section in each repo's `CLAUDE.md`.

**Three cases:**

- **Existing memory policy section** (ban-memory, auto-memory, or similar) ā€”
  replace with the canonical snippet.
- **CLAUDE.md exists, no memory section** ā€” add the canonical snippet after
  any project overview, before development commands.
- **No CLAUDE.md** ā€” create the file with the snippet as initial content.

**Canonical snippet:**

```markdown
## Memory management

Memory is allowed with human approval. The authoritative policy is in
the user's global `~/.claude/CLAUDE.md` ā€” agents must propose memory
writes and suggest a destination (repo memory, global CLAUDE.md, or
plugin/skill issue) before writing. See that file for the full
workflow.

Available skills:
- `/standard-tooling:memory-init` ā€” set up or update the policy header
  in a project's `MEMORY.md`.
- `/standard-tooling:memory-audit` ā€” structured collaborative review
  of memory files.
```

The snippet is identical across all repos ā€” no per-repo customization.

### 3. `block-memory-writes` cleanup

Grep the repo for `block-memory-writes` in active files (CLAUDE.md, hooks
config, live documentation). Remove any references found.

**Excluded from cleanup:** Changelogs, release notes, and historical commit
records. These are auto-generated or historical and should not be edited.

### 4. Audit issue (existing-memory repos only)

Create a GitHub issue in each of the 12 repos with existing memory files.

**Title:** `chore: memory management audit`

**Body contents:**
- Context linking to [#219](https://github.com/wphillipmoore/standard-tooling-plugin/issues/219) (design decision) and [#233](https://github.com/wphillipmoore/standard-tooling-plugin/issues/233) (fleet rollout) ā€” full URLs required since these issues live in `standard-tooling-plugin`, not the consuming repo
- States that the MEMORY.md header and CLAUDE.md snippet are already in place
- Instructs to run `/standard-tooling:memory-audit` interactively
- Lists the number of existing memory files in that repo

No assignee, no milestone.

## Execution Strategy

One subagent per repo. Each subagent:

1. Navigates to the repo at `/Users/pmoore/dev/github/<repo-name>/`
2. Creates a feature branch (`chore/memory-management-policy`)
3. Plants the MEMORY.md header (find existing or create new)
4. Adds or replaces the CLAUDE.md memory management section
5. Greps for `block-memory-writes` in active files and removes references
6. Commits and opens a PR targeting the repo's default branch
7. Creates the audit issue (existing-memory repos only)

**Batching:** 4ā€“5 repos per batch to stay within GitHub rate limits and
manage context.

**PR convention:**
- Title: `chore: add memory management policy`
- Body: links to [#233](https://github.com/wphillipmoore/standard-tooling-plugin/issues/233) (full URL ā€” cross-repo reference), lists what changed and why

## What This Does Not Cover

- **Interactive memory audits** ā€” deferred to per-repo audit issues
- **Changelog or release note edits** ā€” historical records are not touched
- **`standard-tooling-plugin` itself** ā€” changes already shipped with #219
