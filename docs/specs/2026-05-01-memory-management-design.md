# Memory Management — Human-Routed Writes

**Issue:** [#219](https://github.com/wphillipmoore/standard-tooling-plugin/issues/219)
**Date:** 2026-05-01
**Status:** Design approved

## Problem

Agents write to memory autonomously, placing information in the wrong
scope — repo-local when it should be global, or global when it belongs
in a plugin skill. This creates three concrete harms:

1. **Inconsistent fleet behavior.** A behavioral fix recorded in one
   repo's memory does not propagate to the other 15–20 consuming repos.
   The agent that encounters the problem next in a different repo
   re-derives or re-records the same fix locally.

2. **Stale noise accumulation.** Memory files grow with entries that
   were relevant at write time but become outdated as the codebase
   evolves. Without a structured review process, the noise compounds.

3. **Portability violation.** When plugins and skills only work
   correctly because an agent has made notes in memory, the behavior
   breaks for any other user who hasn't accumulated those same notes.
   Software should work from its source, not from ambient agent state.

## Solution

Three reinforcing layers that turn unilateral memory writes into a
human-routed decision.

### Layer 1: Global CLAUDE.md Policy

The authoritative rule, added to `~/.claude/CLAUDE.md`. Applies to
every session regardless of repo.

**Rule:** Never write to, edit, or delete memory files (`MEMORY.md` or
any file under the memory directory) without explicit human approval.

**Workflow when the agent wants to save something:**

1. Tell the human what you would save and why.
2. Suggest where it belongs — one of three destinations:
   - **Repo memory** — specific to this one codebase and only this
     codebase (e.g., a quirk of this repo's test setup).
   - **Global CLAUDE.md** — a cross-repo behavioral preference or
     convention (e.g., preferred error-handling style).
   - **Plugin/skill issue** — about how the standard-tooling suite
     should behave (e.g., a skill needs updating). Create an issue in
     the appropriate repository.
3. Wait for the human to confirm or redirect before taking any action.

**Exemption:** User-invoked skills that write to memory as part of
their documented workflow (e.g., `/handoff stop`) have implicit
approval — the user triggered the skill.

### Layer 2: MEMORY.md Policy Header

An HTML comment placed at the top of every repo's `MEMORY.md` by the
`memory-init` skill. Acts as a trap for agents at the point of write —
self-contained enough to work even without the global policy loaded.

```html
<!-- POLICY: Do not edit this file or any file in this directory without
explicit human approval. Before writing, propose the content to the human
and suggest where it belongs: repo memory, global CLAUDE.md, or a
plugin/skill issue. See the global CLAUDE.md memory management policy
for the full workflow. User-invoked skills (e.g., /handoff) are exempt. -->
```

Format: HTML comment so it is invisible in rendered Markdown but visible
to agents reading raw file content.

Placement: First lines of the file, before any index entries.

### Layer 3: Skills

Two new skills in the `standard-tooling` plugin namespace.

#### `memory-init`

**Invocation:** `/standard-tooling:memory-init`

**Behavior:**

1. Resolve the current project's memory directory path. The path is
   `~/.claude/projects/<slug>/memory/` where `<slug>` is the session's
   starting CWD with `/` replaced by `-` (e.g.,
   `/Users/pmoore/dev/github/myrepo` →
   `-Users-pmoore-dev-github-myrepo`).
2. If the directory does not exist, create it.
3. If `MEMORY.md` does not exist, create it with the policy header and
   an empty index structure.
4. If `MEMORY.md` already exists, check whether the policy header is
   present at the top. If missing or outdated, prepend/update it —
   preserving existing index content.

**Properties:**

- Idempotent — running twice on the same repo is safe.
- Only touches the header; never modifies index entries.
- No category scaffolding (future extension).
- No migration of existing memory files (that is `memory-audit`).

#### `memory-audit`

**Invocation:** `/standard-tooling:memory-audit`

**Workflow:**

1. **Inventory.** Read `MEMORY.md` and list all memory files in the
   directory. Report: file count, types (feedback, project, reference,
   user, handoff), and age (last modified dates). Flag files not indexed
   in `MEMORY.md` and index entries pointing to missing files.

2. **Walk-through.** Present each memory file one at a time. For each:
   - Show name, type, and content (summarized if long).
   - Assess staleness — is this still accurate given current
     code/config?
   - Suggest a disposition with reasoning:
     - **Keep** — still accurate, correctly scoped to this repo.
     - **Update** — substance is right, content needs refreshing.
     - **Relocate to global CLAUDE.md** — cross-repo preference, not
       repo-specific.
     - **Relocate to plugin/skill issue** — about standard-tooling
       suite behavior.
     - **Delete** — stale, redundant, or no longer relevant.

3. **Human decides.** For each file, the human confirms or overrides.
   The agent does not act until the human responds.

4. **Execute.** After the human decides on each file:
   - Keep: no change.
   - Update: edit the file, show the diff.
   - Relocate to global CLAUDE.md: draft the addition, show it, write
     on approval.
   - Relocate to plugin issue: draft the issue body, create on
     approval.
   - Delete: remove the file and its index entry.

5. **Reconcile index.** Update `MEMORY.md` to reflect the final state.
   Ensure the policy header is intact.

**Constraints:**

- Operates on the current project's memory directory only — one repo
  per audit.
- No batch mode — every disposition requires human confirmation.

### Phantom `block-memory-writes` hook — removal

Release notes (v1.3.0, v1.3.1, v1.4.2) reference a
`block-memory-writes` hook that was never implemented. The enforcement
model for this design is prose-based, not hook-based. All references to
this hook must be found and removed:

- Release notes / changelogs
- Any CLAUDE.md files across the fleet
- Any documentation or comments

A grep across the fleet for `block-memory-writes` is part of the
rollout plan.

## Fleet Rollout

After the plugin ships with the new skills and updated policies, every
consuming repo needs attention.

**Per-repo steps:**

1. Run `memory-init` — plant the policy header in MEMORY.md.
2. Run `memory-audit` — review existing memory files accumulated under
   the old unmanaged regime.
3. Update per-repo CLAUDE.md — replace any "ban memory" policy text
   with the new managed approach, or remove it if the global policy and
   MEMORY.md header provide sufficient coverage.

**Mechanism:** Create a GitHub issue in each consuming repo. The issue
describes the three steps, references #219 as the upstream decision,
and provides enough context for a future session to execute without
re-deriving the rationale.

**Scope:** Every repo in the fleet — including those with empty or
nonexistent memory directories, so the policy header is planted
proactively.

**Phantom hook cleanup:** Include a grep across the fleet for
`block-memory-writes` references as part of the rollout issues.
