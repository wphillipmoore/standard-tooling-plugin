# Memory Fleet Rollout Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Plant the memory management policy (CLAUDE.md snippet + MEMORY.md header) across all 20 consuming repos and create audit issues for the 12 repos with existing memory files.

**Architecture:** One subagent per repo, dispatched in parallel batches of 5. Each subagent navigates to the repo, creates a branch, makes the changes, commits, opens a PR, and (for existing-memory repos) creates an audit issue.

**Spec:** [`docs/specs/2026-05-02-memory-fleet-rollout-design.md`](../specs/2026-05-02-memory-fleet-rollout-design.md)

## Canonical Texts

These are the exact strings used by every subagent. Defined once here, referenced by task.

### MEMORY.md Policy Header

```text
<!-- POLICY: Do not edit this file or any file in this directory without
explicit human approval. Before writing, propose the content to the human
and suggest where it belongs: repo memory, global CLAUDE.md, or a
plugin/skill issue. See the global CLAUDE.md memory management policy
for the full workflow. User-invoked skills (e.g., /handoff) are exempt. -->
```

### CLAUDE.md Snippet

```markdown
## Memory management

Memory is allowed with human approval. The authoritative policy is in
the user's global `~/.claude/CLAUDE.md` — agents must propose memory
writes and suggest a destination (repo memory, global CLAUDE.md, or
plugin/skill issue) before writing. See that file for the full
workflow.

Available skills:
- `/standard-tooling:memory-init` — set up or update the policy header
  in a project's `MEMORY.md`.
- `/standard-tooling:memory-audit` — structured collaborative review
  of memory files.
```

### Audit Issue Body Template

```markdown

- MEMORY.md policy header is in place

### Next step

Run `/standard-tooling:memory-audit` in a Claude Code session in this repo.

codebase, assesses staleness, and routes each file to keep/update/relocate/delete
with human approval.

- Design decision: [standard-tooling-plugin#219](https://github.com/wphillipmoore/standard-tooling-plugin/issues/219)
```

## Subagent Prompt Template

Each subagent receives a filled-in version of this prompt. The orchestrator
substitutes the `{PLACEHOLDERS}` from the reference table above.

```text
You are rolling out the memory management policy to the repo `{REPO_NAME}`.

## Context

This is part of a fleet-wide rollout (standard-tooling-plugin#233) that
plants the three-layer memory management defense across all consuming repos.
You are handling one repo.

## Your repo

- Path: /Users/pmoore/dev/github/{REPO_NAME}/
- Default branch: {DEFAULT_BRANCH}
- CLAUDE.md action: {CLAUDE_ACTION}
- MEMORY.md action: {MEMORY_ACTION}
- Create audit issue: {YES_OR_NO}
- Memory file count: {MEMORY_COUNT}

## Step 1: Create a feature branch

cd /Users/pmoore/dev/github/{REPO_NAME}/ && git checkout {DEFAULT_BRANCH} && git pull && git checkout -b chore/memory-management-policy

## Step 2: Update MEMORY.md

Memory path: /Users/pmoore/.claude/projects/-Users-pmoore-dev-github-{REPO_NAME}/memory/MEMORY.md

**If MEMORY.md exists with a BANNED header** ("DO NOT USE THIS FILE" or
similar ban text): Replace ONLY the ban header/comment with the canonical
policy header. Preserve all index entries below.

**If MEMORY.md exists with real index content** (no ban header): Prepend the
canonical policy header above the existing content. If there's already a
policy header (starts with `<!-- POLICY:`), replace it.

**If MEMORY.md does not exist**: Create the directory (mkdir -p) and write
MEMORY.md with just the policy header.

Canonical policy header:

<!-- POLICY: Do not edit this file or any file in this directory without
explicit human approval. Before writing, propose the content to the human
and suggest where it belongs: repo memory, global CLAUDE.md, or a
plugin/skill issue. See the global CLAUDE.md memory management policy
for the full workflow. User-invoked skills (e.g., /handoff) are exempt. -->

## Step 3: Update CLAUDE.md

**If CLAUDE.md has a ban-memory section** (contains "Do NOT use MEMORY.md",
"Auto-memory policy", or similar ban language): Find that section and replace
it with the canonical snippet below. Keep all other sections intact.

**If CLAUDE.md exists but has no memory section**: Add the canonical snippet.
Place it after any project overview / introductory section, before
development commands or technical sections. Use your judgment on placement —
it should feel natural in the document flow.

**If no CLAUDE.md exists**: Create CLAUDE.md with a `# CLAUDE.md` title line
followed by a blank line and the canonical snippet.

Canonical snippet:

## Memory management

Memory is allowed with human approval. The authoritative policy is in
the user's global `~/.claude/CLAUDE.md` — agents must propose memory
writes and suggest a destination (repo memory, global CLAUDE.md, or
plugin/skill issue) before writing. See that file for the full
workflow.

Available skills:
- `/standard-tooling:memory-init` — set up or update the policy header
  in a project's `MEMORY.md`.
- `/standard-tooling:memory-audit` — structured collaborative review
  of memory files.

## Step 4: Verify no block-memory-writes references

Run: grep -r "block-memory-writes" /Users/pmoore/dev/github/{REPO_NAME}/ --include="*.md" --include="*.json" --exclude-dir=.git | grep -v CHANGELOG | grep -v releases/

If any hits in active files (CLAUDE.md, hooks config, live docs), remove them.
If no hits, move on.

## Step 5: Commit and open PR

Stage all changed files (CLAUDE.md only — MEMORY.md is outside the repo).
Commit with message: "chore: add memory management policy"
Push the branch and open a PR targeting {DEFAULT_BRANCH}.

PR title: chore: add memory management policy

Write the PR body to a temp file and pass it via --body-file:

## Summary

Plant the memory management policy across the fleet per
[standard-tooling-plugin#233](https://github.com/wphillipmoore/standard-tooling-plugin/issues/233).

Changes:
- Added/updated MEMORY.md with the canonical policy header
- Added/updated CLAUDE.md with the memory management snippet
- Verified no `block-memory-writes` references in active files

IMPORTANT: Do NOT use heredocs for the PR body. Write it to a temp file
and use --body-file.

## Step 6: Create audit issue (only if {CREATE_AUDIT_ISSUE} is Yes)

Create a GitHub issue in repo wphillipmoore/{REPO_NAME}.

Title: chore: memory management audit

Write the issue body to a temp file and pass it via --body-file to
gh issue create. The body:

## Memory management audit

The memory management policy has been rolled out to this repo:
- MEMORY.md policy header is in place
- CLAUDE.md memory management snippet is in place

This repo has **{MEMORY_COUNT}** existing memory files that need interactive review.

### Next step

Run `/standard-tooling:memory-audit` in a Claude Code session in this repo.
The audit walks through each memory file, verifies claims against the
codebase, assesses staleness, and routes each file to keep/update/relocate/delete
with human approval.

### Context

- Design decision: [standard-tooling-plugin#219](https://github.com/wphillipmoore/standard-tooling-plugin/issues/219)
- Fleet rollout: [standard-tooling-plugin#233](https://github.com/wphillipmoore/standard-tooling-plugin/issues/233)

## Deliverables

When done, report:
1. PR URL
2. Audit issue URL (if created)
3. Any block-memory-writes references found (expected: none)
4. Any problems encountered
```
